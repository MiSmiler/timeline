# Grill for Good Day

通过追问帮助用户进行日常复盘和未来规划。

## 工作流程

1. **读取数据**
   - 读取今天的 `timelines/YYYY-MM-DD.md`
   - 读取近几天的 timelines（了解上下文）
   - 读取 `plans/` 目录下的所有计划文件
   - 读取 `reviews/` 目录了解历史复盘（如果存在）

2. **追问对话**
   按以下顺序逐一提问，每次只问一个问题：

   a. **今日回顾** → 基于今天的 Events 和 Todos
      - 参考具体事件提问：「今天做了 X，进展如何？」
      - 参考未完成计划提问：「计划做 Y 但没完成，遇到什么障碍？」

   b. **短期计划对齐** → 对照 `plans/` 中的相关计划
      - 「今日做的事是否在推进短期目标？」
      - 「短期目标进展如何？需要调整吗？」

   c. **长期计划对齐** → 对照 `plans/` 中的长期规划
      - 「短期目标是否在服务长期愿景？」
      - 「长期目标是否需要调整？」

3. **生成输出**
   追问结束后，一次性写入以下文件：

   - `reviews/YYYY-MM-DD_review.md`（今日复盘）
   - 更新 `plans/` 中相关文件（如有调整）

## 文件格式

### Review 文件

路径：`reviews/YYYY-MM-DD_review.md`

```markdown
# Review: YYYY-MM-DD

## 今日概括
（总结做了什么，提炼而非搬运 timelines 数据）

## 明日计划
（追问后确定的明天重点）

## 反思
（对话中产生的感悟、改进点）
```

### Plans 目录

路径：`plans/`

按主题组织，每个主题一个文件（如 `work.md`、`learning.md`、`personal.md`）。

文件内部按子主题用章节区分：
```markdown
# Work

## 子主题A
- ...

## 子主题B
- ...
```

## 追问原则

- 每次只问一个问题，等待用户回答
- 基于具体数据提问，不泛泛而问
- 追问形成链路：Events → 今日计划 → 短期计划 → 长期计划
- 引导用户思考，而非替用户做决定
- 不自动写入，追问结束后一次性生成输出
- **保持与用户一致的复盘粒度**：如果用户明确说按「日」复盘，不要擅自拉到「周」或「月」的视角。追问中的时间参照（如「上周的目标」「两周冲刺进展」）应聚焦到当天做了什么，而非跨越更长周期评估进度

## 常见场景

### 深夜晚间复盘（00:00-05:00 触发）
- 此时用户通常在回顾「昨天」，review 文件应归属昨天的日期
- 时间参照保持在当天范围内，不要跳跃到周/月视角
- 「明日计划」指的是天亮后醒来的一天
