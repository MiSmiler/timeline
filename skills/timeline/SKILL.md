# Timeline 技能

管理存储在 markdown 文件中的待办事项（Todo）和事件（Event），按日期组织。

## 数据存储

### 文件位置
- 目录：项目根目录下的 `timelines/`
- 文件名格式：`YYYY-MM-DD.md`（如 `2026-06-12.md`）
- 无日期待办：`0000-00-00.md`（**仅存储 Todos**，不允许 Events 和 Notes 章节，且所有 todo 不能有时间前缀）

### 文件结构
每个文件遵循以下模板：

```markdown
# YYYY-MM-DD

## Events

## Todos

## Notes
```

**例外：`0000-00-00.md` 只有 `## Todos`：**

```markdown
# 0000-00-00

## Todos
```

### 数据格式

#### Events（事件）
- 格式：`- HH:MM 描述。`
- 必须有时间
- 表示在特定时刻发生的事情
- 支持详情（缩进 4 空格）：

```markdown
- 14:30 和客户开会。
    - 讨论了项目时间线。
```

**格式原则：主句精简，细节折叠**
- 主句只保留核心动作（谁做了什么），尽量一行以内
- 补充信息、背景、结果等放到缩进详情里
- 用户偏好简洁的事件列表，不要把所有细节堆在主句中

```markdown
# ✅ 好的格式
- 14:30 和张三开会讨论新项目。
    - 确定了第一阶段的截止日期。
    - 张三负责前端，我负责后端。

# ❌ 不好的格式
- 14:30 和张三开会讨论新项目，确定了第一阶段的截止日期，张三负责前端，我负责后端。
```

#### Todos（待办）
- 格式：`- [ ] HH:MM 描述`（有时间）或 `- [ ] 描述`（无时间）
- 状态：
  - `[ ]` - 未完成
  - `[x]` - 已完成
  - `[ ] ~~描述~~` - 已放弃（删除线）
- 支持详情：

```markdown
- [ ] 15:00 收拾桌面
    - 扔掉过期零食。
```

#### Notes（笔记）
- 自由格式文本，记录想法、目标、心情等
- 无特定格式要求

### 排序规则
- 每个章节内，有时间的条目按时间排序
- 无时间的条目排在最后

## 操作流程

### 预检查

**在任何涉及日期/时间的操作前，务必先检查当前系统时间。**

1. 检查当前时间
2. 如果时间在 00:00-05:00（深夜/凌晨）：
   - 用户说"今天"可能指昨天（还没睡觉）
   - 用户说"明天"可能指今天
   - **务必确认**用户实际指哪天
3. 确认日期后再继续操作

### 创建待办
**触发词**：「提醒」「待办」「要做」「创建待办」

**示例**：
- "提醒我明天三点开会" → 创建明天 15:00 的待办
- "创建待办，明天买洗发水" → 创建明天的待办（无时间）
- "下周三提醒我准备会议纪要" → 创建下周三的待办

**日期解析**：
- 相对：「今天」「明天」「后天」
- 具体：「下周一」「这周五」「6月15日」
- 不确定时，主动询问

**时间解析**：
- 默认：下午（如"三点" → 15:00）
- 「凌晨」前缀表示凌晨时段
- 深夜/凌晨时（午夜后），"明天"可能指"今天"——需确认

**Cron 提醒**：
- 如果待办是**今天**的且有时间，创建对应的 cron job
- 命名格式：`timeline-{YYYYMMDD}-{HHMM}`（如 `timeline-20260612-1500`）
- 如果该时间已有 cron job（同一天多个待办），不重复创建
- 使用 `todo_by_time.sh`（wrapper）作为 cron 的脚本，no_agent 模式运行

### 创建事件
**触发词**：「约了」「买了」等完成动作的动词，以及「创建成事件」「记一下」「记录事件」

**重要**：事件时间 = 动作发生的时间，不是预定事件的时间。

**边界**：用户说「写 log」「写日记」时，是用户自己在写，agent 不要主动代写。如实记录事件即可。

**示例**：
- "约了客户明天下午三点开会" → 事件创建为当前时间，记录你约了这个会
- "买了洗发水了" → 事件创建为当前时间
- "当前正在处理timeline项目" + "创建成事件" → 事件创建为当前时间

**追补模式**：用户先随口说了某件事，随后要求"创建成事件"。此时事件时间取当前时间。

**创建未来承诺的事件后**，询问用户是否要创建对应的待办。

**时间规则**：
- 必须有时间
- 未指定则默认当前时间
- 可以指定过去的时间

### 完成待办
**触发词**：「做完了」「搞定了」「已完成」「做完了」等完成动作的动词

**流程**：
1. 创建一个 Event 记录完成的动作
2. 在所有文件中搜索匹配的 Todo（包括 `0000-00-00.md`）
3. 如果恰好匹配一条：标记为 `[x]`
4. 如果匹配多条：询问用户选哪条
5. 如果没有匹配：只创建 Event

**Cron 清理**：
- 完成待办后，检查该时间是否还有其他未完成待办
- 如果没有，删除对应的 cron job
- 使用 cronjob 工具的 remove 操作

**示例**：
```
用户：买了洗发水了
AI：
- 创建 Event："15:30 买了洗发水。"
- 搜索匹配 Todo："买洗发水"
- 标记 Todo 为 [x]
- 检查是否需要删除 cron job
```

### 放弃待办
**触发词**：「不做了」「算了」「取消」「放弃」

**操作**：给待办描述加删除线
- `- [ ] ~~描述~~`

**Cron 清理**：同「完成待办」流程。

### 移动待办
**触发词**：「移到」「推到」「延期到」「改到」

**示例**：
- "把这个移到明天" → 移到明天
- "推到晚上六点" → 改时间为 18:00（同一天或目标天）
- "昨天的待办移到今天" → 把昨天的待办移到今天

**Cron 处理**：
- 如果原时间的 cron job 没有其他待办，删除旧 cron
- 如果新时间是今天，创建新 cron job
- 命名格式：`timeline-{YYYYMMDD}-{HHMM}`

### 编辑
**触发词**：「改成」「修改」「换成」「改为」「改名」

**支持**：修改 Todo 或 Event 的时间或描述

**Cron 处理**：
- 如果修改了时间：同「移动待办」流程（删旧 cron、建新 cron）
- 如果只修改描述（时间不变）：不操作 cron

### 删除待办
**触发词**：「删掉」「删除」「可以删掉」

**操作**：删除前先确认。但如果用户明确说"删掉"或"可以删掉"，意图清晰，可直接执行无需二次确认。

**流程**：
1. 在所有文件中搜索匹配的 Todo（包括 `0000-00-00.md`）
2. 如果匹配多条：询问用户选哪条
3. 删除匹配的 Todo 行
4. Cron 清理：检查该时间是否还有其他未完成待办，如果没有则删除对应的 cron job

**Cron 清理**：同「完成待办」流程。

### 查看
**触发词**：「今天有什么事」「看看今天的待办」「今天的事」

**操作**：显示指定日期的 Events、Todos 和 Notes（默认：今天）

### 回顾
**触发词**：「有什么事做」「过期的」「看看没做完的」「当前漏做了什么」「帮我看看还有什么没做的」

**操作**：显示所有过期的未完成待办

**快捷方式**：可使用脚本 `scripts/todo_overdue.py` 获取过期和无时间待办。

### 补充详情
**触发词**：「补充」「备注」「加个备注」

**操作**：给现有条目添加缩进的详情

**也可以在创建时添加详情**：
- "提醒我三点收拾桌面，记得扔掉过期零食"

## 辅助脚本

### 列出所有未完成待办
```bash
python3 scripts/todo_list.py
```

### 按时间提取待办
```bash
python3 scripts/todo_by_time.py [YYYY-MM-DD HH:MM]
```
用于 cron job 定时提醒，有则输出，无则静默。不传参数则使用当前系统时间。

### 提取过期和无时间待办
```bash
python3 scripts/todo_overdue.py [YYYY-MM-DD HH:MM]
```
用于每小时扫描，有则输出，无则静默。不传参数则使用当前系统时间。

### 验证日期一致性
```bash
python3 scripts/validate.py
```
检查 H1 日期是否与文件名匹配。

## Cron 生命周期管理

以下规则控制何时创建/删除 cron job：
| 操作 | Cron 处理 |
|------|-----------|
| 创建今天有时间的待办 | 检查是否已有同名 cron，无则创建 |
| 修改待办时间 | 删旧 cron（若无其他待办），建新 cron（若新时间是今天） |
| 完成/取消待办 | 检查该时间是否还有其他待办，无则删 cron |
| 删除待办 | 同上 |
| 创建非今天的待办 | 不创建 cron（由 daily review 负责） |

## 时间解析规则

| 用户说法 | 解释 |
|---------|------|
| "三点" | 15:00（默认下午） |
| "凌晨三点" | 03:00 |
| "早上九点" | 09:00 |
| "今晚八点" | 20:00 今天 |
| "明天" | 明天 |
| "下周三" | 下周三 |

**深夜边界**：当前时间在午夜后，用户说"明天"时，确认是指"今天"（还没睡）还是真正的明天。

**模糊情况**：务必询问确认（如"下周开会" → "请问是下周几？"）

---

## 边界情况

1. **完成待办时没有匹配**：只创建 Event，告知用户
2. **多个可能匹配**：列出选项让用户确认
3. **新待办的时间在过去**：允许（用于记录遗漏的事项）
4. **文件不存在**：用标准模板创建
5. **空章节**：始终预创建三个 H2 标题（Events、Todos、Notes）；`0000-00-00.md` 例外，只预创建 `## Todos`

## 使用模式

### 实时日记模式
用户可能在整个晚上/一天中随时报告事件，期望快速、简洁的记录。典型对话流：
```
用户：开始做饭        → 记录 Event，简短回复
用户：煮完饭了，开吃  → 记录 Event，简短回复
用户：吃完饭了        → 记录 Event，简短回复
用户：开始洗碗        → 记录 Event，简短回复
```
**要点**：回复保持极简（表情 + 内容），不要每次都展示完整文件内容。用户在快速记录时不需要冗长确认。
